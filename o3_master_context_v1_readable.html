Section 01

Executive Summary

O³ ZONE คือ AI-powered Workforce Intelligence Platform ที่ช่วย SME ไทยและองค์กรที่กำลังเติบโต เปลี่ยนข้อมูลบริษัทและ HR พื้นฐาน ให้กลายเป็น Workforce Insights, Business Recommendations และ Action Plans ที่นำไปใช้ได้จริง

O³ ไม่ใช่แค่ Dashboard, ไม่ใช่ HRIS, ไม่ใช่ Chatbot — แต่เป็น Platform ที่รวม Data, AI, Insight Engine, Academy และ Benchmark Center ไว้ด้วยกันบนรากฐานเดียวกัน

Founder Note

O³ ถูกออกแบบให้ SME ไทยใช้งานได้ภายในไม่กี่นาที — อัปโหลดไฟล์ Excel ไฟล์เดียว ระบบจะตรวจสอบคุณภาพข้อมูล สรุปภาพรวมกำลังคน ชี้ความเสี่ยง แนะนำ Action Plan และให้ AI อธิบายทุกอย่างเป็นภาษาไทยที่เข้าใจง่าย

Section 02

Why This Document Exists

O³ ZONE เป็น Platform ที่ซับซ้อนระดับหนึ่ง การตัดสินใจเรื่อง Architecture, Data Model, AI Strategy และ Product Scope ต้องมี แหล่งความจริงเพียงหนึ่งเดียว ที่ทุกคน — Founder, Developer, AI Agent, Product Team — ใช้อ้างอิงร่วมกัน

เอกสารนี้คือ Master Context ที่รวบรวมทุกการตัดสินใจสำคัญของ O³ ไว้ในที่เดียว ก่อนที่จะแตกออกเป็น Handbook, Product Spec, Developer Spec, Prototype หรือ Code จริง

Important Decision

Documentation และ Architecture ต้องมาก่อนการเขียนโค้ดเสมอ — นี่คือ Principle 14 ของ O³ Platform Constitution

Section 07

Platform Principles (15 Principles)

หลักการ 15 ข้อที่เป็นรากฐานของทุกการตัดสินใจด้าน Architecture, Product และ Engineering ใน O³ ZONE

Key Principle 01

One Source of Truth — ทุก Business Object ต้องมีแหล่งข้อมูลที่เชื่อถือได้เพียงหนึ่งเดียว Employee Master ต้องไม่ถูกทำซ้ำใน Dashboard, Survey, Academy และ AI Tools

Key Principle 02

Everything Starts from OWDS — O³ Workforce Data Standard คือภาษากลางสำหรับ Data Ingestion, Dashboard, AI, Survey และ Benchmark

Key Principle 03

API First — ทุก Product ต้องสื่อสารผ่าน API เท่านั้น Frontend ต้องไม่ต่อ Database โดยตรง AI ต้องไม่เรียก LLM โดยไม่ผ่าน AI Gateway

Key Principle 04

Insight First, Report Second — O³ ให้ความสำคัญกับการตีความทางธุรกิจ ความเสี่ยง ลำดับความสำคัญ และ Action Plan — ไม่ใช่แค่การแสดง Report

Key Principle 05

AI Must Explain — ทุก AI Recommendation ต้องระบุ เหตุผล หลักฐาน ระดับความเชื่อมั่น และข้อจำกัด

Section 08

Domain Model

ก่อนออกแบบ Database หรือ API — O³ ต้องกำหนด Business Objects ที่ประกอบกันเป็น Platform ก่อน นี่คือ Domain Model ที่อธิบายโลกของ O³ ในภาษาธุรกิจ

🏢 Company

Root Domain — เป็นเจ้าของ Users, Employees, Subscription และ Product Usage ทั้งหมด

👤 Employee

Core Workforce Object — เชื่อมกับ Performance, Compensation, Training, Exit, Survey

📊 Insight

Interpreted Layer — แปลข้อมูลดิบเป็นความหมายทางธุรกิจ ความเสี่ยง และลำดับความสำคัญ

🤖 AI Advisor

ไม่ใช่ Data Owner — อ่าน Context และสร้าง Recommendations จาก OWDS + Semantic Layer

📈 Benchmark

Anonymized Aggregated Data — ไม่ได้สร้างจาก Individual Records ที่ระบุตัวตนได้

Section 09

OWDS — O³ Workforce Data Standard

OWDS คือมาตรฐานข้อมูลกลางที่ทำให้ O³ สามารถอ่านข้อมูลบริษัทจาก Excel, CSV, API, HRIS หรือ Manual Input และแปลงเป็นโครงสร้างเดียวกันสำหรับ Dashboard, AI, Survey และ Benchmark

ADR-001: Use OWDS as Standard

หากไม่มี OWDS ข้อมูลลูกค้าทุกรายจะแตกต่างกัน — "Name" vs "Employee Name" vs "Staff Name" — ทำให้เกิดปัญหา Mapping และการวิเคราะห์ที่ไม่สอดคล้องกัน OWDS กำหนด Standard Fields เช่น Employee_ID, Department, Position, Salary, Start_Date, Exit_Date, Performance_Rating, Potential

One Simple Company Data Template (7 Sheets)

Sheetเนื้อหาใช้สำหรับ
1. Company_Profileข้อมูลบริษัท, Industry, SizeContext สำหรับ AI และ Benchmark
2. Employee_Masterรายชื่อพนักงาน, Department, Position, SalaryWorkforce Snapshot, Headcount
3. Exit_Recordการลาออก, เหตุผล, Regrettable LossTurnover, Attrition Cost
4. PerformancePerformance Rating, Potential, Key TalentTalent & Performance Dashboard
5. Trainingชั่วโมงอบรม, Category, CostLearning & Development
6. Business_OutputRevenue, Output, ProductivityProductivity Dashboard
7. Short_Employee_SurveyEngagement, Workload, SatisfactionEmployee Sentiment

Regrettable Loss Logic

การลาออกถูกจัดเป็น Regrettable Loss เมื่อเข้าเงื่อนไขข้อใดข้อหนึ่ง:

  • Performance Rating สูง
  • Potential สูง
  • เป็น Key Talent
  • อยู่ใน Critical Position
  • Replacement Difficulty สูง
  • อยู่ใน Strategic Role

AI Guardrails

4. Rules for AI Agents

กฎที่ AI Agent ทุกตัวต้องปฏิบัติตามเมื่อทำงานกับ O³ ZONE:

Critical Rules
#Ruleคำอธิบาย
1Do not create Product Specs before reading the Master ContextProduct Specs ต้องอ้างอิง Domain Model, OWDS และ Architecture ที่กำหนดไว้แล้ว
2Do not invent new product namesใช้ชื่อที่กำหนดใน Master Context เท่านั้น: O³ Dashboard, O³ Workforce AI Studio, O³ Survey Studio, O³ Academy, O³ Benchmark Center
3Use O³ vocabulary consistentlyใช้คำศัพท์ตาม Vocabulary/Glossary — Workforce Intelligence, Insight, Action Plan, AI Advisor, OWDS, Semantic Layer
4Markdown is the source of truthเอกสารทั้งหมดต้องเขียนเป็น Markdown ก่อน — HTML สร้างจาก Markdown ทีหลัง
5HTML is generated from Markdownห้ามสร้าง HTML โดยไม่มี Markdown ต้นทาง
6Every Book must reference related Booksทุก Book ต้องมี Cross References ไปยัง Books ที่เกี่ยวข้อง
7Every technical decision must connect to an ADRการตัดสินใจด้านเทคนิคที่สำคัญต้องมี ADR รองรับ
8Every data concept must connect to OWDSแนวคิดด้านข้อมูลทุกอย่างต้องอ้างอิง OWDS
9Every KPI must connect to the Semantic Layerทุก KPI ต้องมี Definition, Formula, Interpretation ใน Semantic Layer
10Every AI feature must explain its reasoningAI ต้องอธิบาย Summary, Evidence, Interpretation, Action, Confidence, Limitations
11If data is insufficient, AI must say soAI ต้องไม่เดา — ต้องระบุว่าข้อมูลอะไรขาด และอะไรที่ยังพอสรุปได้
12Do not skip Domain Model, OWDS, API, or AI Architecture4 เสาหลักนี้ต้องมาก่อนการสร้าง Product Specs หรือ Code เสมอ
